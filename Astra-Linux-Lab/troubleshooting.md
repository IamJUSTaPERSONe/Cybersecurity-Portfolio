# Проблемы, возникшие в ходе выполнения работы, и способы их решения

## Проблема 1: Неверная директива local в конфиге
**Симптомы**: Сервер не запускался через systemd, в логах ошибка Options error.

**Решение**: В конфигурационном файле (.ovpn) была обнаружена устаревшая строка local 10.0.2.15,
указывающая на неактуальный IP из NAT-сети. После её удаления сервер успешно запустился и начал слушать порт 1194.

## Проблема 2: Несовместимость шифрования (ГОСТ vs AES)
**Симптомы**: Клиент доходил до сервера, но на этапе TLS handshake происходил сбой.

**Решение**: На сервере по умолчанию использовался алгоритм "Кузнечик" (grasshopper-cbc),
который клиент OpenVPN на Windows не поддерживает. Замена шифрования на универсальный AES-256-GCM как в конфиге сервера, так и в клиентском файле решила проблему.

## Проблема 3: Путаница с путями к сертификатам
**Симптомы**: Клиентские сертификаты, созданные вручную через easy-rsa, не подходили, так как были подписаны другим CA.

**Решение**: Использование готовых файлов из директории, созданной графической утилитой (/etc/openvpn/openvpn-certificates/keys/). Именно там лежали правильные, корректно подписанные сертификаты.

## Проблема 4: Ошибка подключения к серверу при раздаче интернета через телефон
**Симптомы**: При раздаче интернета на хост с помощью кабеля или точки доступа с телефона клиент не видел в сети виртуальную машину, так как
1. Телефонная точка доступа создает "изолированную" сеть
2. При раздаче интернета с телефона он сам назначает IP (Телефон раздает сеть другого диапазона (скорее всего 192.168.43.x), а у сервера статический адрес из диапазона 192.168.0.x.),
3. Между устройствами в этой сети часто блокируется обмен (клиентская изоляция),
4. Если интернет на хосте, но пинг 8.8.8.8 из виртуалки не идет — значит VirtualBox не может "вытащить" интернет в виртуалку через телефонное подключение.

**Решение**: Либо настраивать сеть специально под интернет от телефона (негарантированно), либо подключиться к домашней сети от роутера.
```
# На хосте (Windows) проверить свой IP
ipconfig

# В виртуалке проверить, есть ли интернет
ping 8.8.8.8

# Проверить, видит ли виртуалка хост
ping <IP_хоста>
```

## Проблема 5: Конфликт запуска через командную строку и графическую утилиту
**Симптомы**: Сервер запускался графической утилитой и имел статус "Сервер запущен",
но systemd через командную строку показывал статус сервера dead (sudo systemctl status openvpn-server@server). 

**Решение**: Использование только штатных средств Astra Linux (fly-admin-openvpn-server) для управления сервером,
отказ от смешивания способов запуска. Это особенность архитектуры Astra Linux: графическая утилита и systemd — два независимых способа управления

При запуске через графическую утилиту происзодит:
- OpenVPN запускается как процесс с PID
- Использует конфигурацию, созданную утилитой
- Не регистрируется в systemd как служба
- Управляется через графический интерфейс
При запуске через systemd (sudo systemctl start openvpn-server@server):
- Запускается системная служба
- Использует стандартный конфиг /etc/openvpn/server.conf
- Управляется через systemd
  
✔️ Для проверки состояния в командной строке можно использовать
```
# Проверка, прослугивается ли данный порт сервером
sudo netstat -anp | grep 1194
```
