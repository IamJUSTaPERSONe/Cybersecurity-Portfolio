# Анализ инцидента: Path Traversal
```
192.168.17.22 - - [28/Apr/2026:02:15:33 +0300] "GET /index.php?page=../../../../etc/passwd HTTP/1.1" 404 521 "-" "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36"
```
## Разбор лога

| Время | IP источника | HTTP метод | Запрос | Код ответа |
|-------|--------------|------------|--------|-------------|
| 28/Apr/2026:02:15:33 | 192.168.17.22 | GET | `/index.php?page=../../../../etc/passwd` | 404 |

**User-Agent:** `Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36`

## Анализ

Запрос содержит конструкцию `../../../../etc/passwd` — это классический признак атаки **Path Traversal**.  
Злоумышленник пытается переместиться по директориям вверх и прочитать системный файл с информацией о пользователях Linux.

**Причина подозрения:**  
- Время запроса — 2:15 ночи (нерабочее время)
- User-Agent не скрыт, что может говорить об автоматизированном сканере или запуске атаки через утилиту типа `curl`

**Результат атаки:**  
Сервер вернул ошибку 404 — файл не найден или доступ запрещен. Атака не удалась.

## MITRE ATT&CK Mapping

| Этап атаки | Тактика (Tactic) | Техника (Technique) |
|------------|------------------|----------------------|
| Злоумышленник уже внутри разрешенного трафика (HTTP) и осматривает файловую систему | **Discovery** (ТА0007) | **T1083** — File and Directory Discovery |

**Почему Discovery, а не Credential Access (TA0006)?**  
Потому что в этом логе нет прямых действий по краже паролей (например, брутфорса или кейлоггера). Злоумышленник только «прощупывает» систему: какие файлы доступны, как устроены пути. Это типичная разведка внутри сети после первоначального проникновения.  
**TA0006** описывает уже финальные методы получения учетных данных, а **TA0007** — сбор информации о системе перед принятием решения о следующих шагах атаки.

## Выводы

- Атака Path Traversal обнаружена и идентифицирована.
- Сервер защищен от данной уязвимости (ответ 404).
- Злоумышленник находится на стадии внутренней разведки (Discovery), что требует усиления мониторинга.

## Рекомендации

1. Продолжать мониторинг веб-логов на подобные паттерны (`../`, `passwd`, `boot.ini`).
2. Настроить алерт в SIEM на комбинацию `GET` + `../` + `etc/passwd`.
3. Рассмотреть возможность добавления WAF (Web Application Firewall) для автоматической блокировки таких запросов.
