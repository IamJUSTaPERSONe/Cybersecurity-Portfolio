
# Основы журналирования и SIEM

## 1. Сетевые устройства и источники журналов

В любой сети несколько устройств взаимодействуют друг с другом и с Интернетом (как правило, через маршрутизатор). К таким устройствам относятся серверы данных, конечные точки на базе Linux/Windows и т.д.

**Эти устройства постоянно ведут журналы (логи)** — записи о происходящих внутри них событиях. Поэтому их также называют **источниками журналов**.

Логи делятся на две основные категории:

### 1.1 Host-Centric Log Sources (Ориентированные на хост)
Эти источники фиксируют события, происходящие *внутри* самого хоста или связанные с ним.
*   **Устройства:** Windows, Linux, серверы.
*   **Примеры событий:**
    *   Пользователь открывает файл.
    *   Попытка аутентификации пользователя.
    *   Запуск процесса.
    *   Изменение или удаление ключа реестра.
    *   Выполнение команд PowerShell.

### 1.2 Network-Centric Log Sources (Сетевые источники)
Эти журналы создаются при взаимодействии хостов друг с другом или при подключении к внешним ресурсам.
*   **Устройства:** Брандмауэры (межсетевые экраны), IDS/IPS, маршрутизаторы.
*   **Примеры событий:**
    *   Подключение по SSH.
    *   Доступ к файлу по FTP.
    *   Обычный веб-трафик.
    *   Подключение к VPN.
    *   Обмен файлами по сети.

> В совокупности эти источники ежесекундно генерируют огромное количество данных.

## 2. Проблемы анализа журналов

Несмотря на обилие данных, анализ логов напрямую с устройств сопряжен с серьезными трудностями:

1.  **Множество источников:** Сотни событий в секунду хранятся на разных устройствах. Изучать их по одному вручную — крайне утомительно.
2.  **Отсутствие централизации:** Для анализа приходится подключаться к каждой машине отдельно (по SSH, RDP и т.д.), что очень неэффективно.
3.  **Ограниченный контекст:** Одна запись в логе может выглядеть безобидно. Лишь сопоставив её с другими событиями (например, доступ к файлу после взлома соседнего ПК), можно увидеть полную картину инцидента.
4.  **Ограниченный анализ:** Вручную проанализировать тонны логов в реальном времени практически невозможно. Аналитики неизбежно пропускают важные аномалии.
5.  **Проблемы с форматом:** Разные источники используют разные форматы данных. Знание всех форматов требует высокой квалификации и замедляет работу.

## 3. Решение: SIEM (Security Information and Event Management)

**SIEM (Система управления информационной безопасностью и событиями безопасности)** — это решение, которое собирает логи из различных источников, стандартизирует их, сопоставляет и выявляет вредоносные действия с помощью правил обнаружения.

### Ключевые возможности SIEM

*   **Централизованный сбор журналов:**
    Логи собираются со всех终ечных точек, серверов, брандмауэров в одном месте с помощью легковесных агентов или API. Это решает проблему необходимости заходить на каждую машину по отдельности.

*   **Нормализация журналов:**
    Система приводит все логи к единому формату.
    *   **Парсинг (разбор):** Разделение сырого лога на отдельные поля (например, `timestamp`, `ip_address`, `username`).
    *   **Нормализация:** Преобразование данных из разных источников в унифицированную структуру. Благодаря этому правилу гораздо легче работать с данными, независимо от их происхождения.

*   **Корреляция журналов:**
    SIEM связывает события из разных источников в единую цепочку. Это позволяет увидеть закономерности, которые не видны при изолированном анализе.
    *   *Пример:* Харис заходит по VPN с нового IP -> Харис открывает документы на общем диске -> Харис запускает PowerShell -> Система инициирует исходящее соединение. По отдельности действия нормальны, но вместе они могут указывать на утечку данных из-за скомпрометированного VPN.

*   **Оповещения в реальном времени:**
    В SIEM загружены правила обнаружения (встроенные или созданные аналитиками). При совпадении условий правила срабатывает **алерт (оповещение)**. Аналитики получают уведомление и могут немедленно приступить к расследованию.

*   **Дашборды и отчеты:**
    SIEM визуализирует данные после обработки. Дашборды предоставляют "быстрые" метрики для мониторинга.
    *   *Примеры виджетов на дашборде:*
        *   Топ предупреждений
        *   Системные уведомления
        *   Неудачные попытки входа
        *   Количество обработанных событий
        *   Самые посещаемые домены

## 4. Источники журналов и методы их сбора

### Как выглядят логи популярных систем?

*   **Windows:**
    Все события имеют уникальный **Event ID** (Идентификатор события). Просмотр осуществляется через оснастку **Event Viewer**. Эти логи пересылаются в SIEM для мониторинга.

*   **Linux:**
    Логи хранятся в текстовых файлах в каталоге `/var/log/`.
    *   `/var/log/httpd/` — логи веб-сервера Apache.
    *   `/var/log/cron` — события планировщика заданий cron.
    *   `/var/log/auth.log` или `/var/log/secure` — логи аутентификации.
    *   `/var/log/kern` — события ядра.

*   **Веб-сервер (Apache):**
    Содержит информацию о входящих HTTP-запросах, IP-адресах клиентах, агентах пользователя (браузерах) и кодах ответа.

### Методы сбора логов в SIEM

1.  **Агент/Переадресатор (Agent/Forwarder):** Легковесная программа, устанавливаемая на конечную точку. Она собирает логи и отправляет их на SIEM-сервер.
2.  **Syslog:** Стандартный протокол для отправки логов в реальном времени от различных систем (серверы БД, сетевые устройства) в централизованное хранилище.
3.  **Ручная загрузка:** Возможность загрузить файлы с логами вручную для разового анализа (полезно в расследованиях).
4.  **Перенаправление портов:** Настройка SIEM на прослушивание определенного порта для приема данных от конечных устройств.

## 5. Механизм срабатывания оповещений

### Правила обнаружения (Detection Rules)

Оповещения создаются на основе **правил обнаружения**. Правило — это логическое выражение, которое срабатывает при выполнении заданных условий.

*   **Примеры правил:**
    *   ЕСЛИ `пользователь совершил >5 неудачных попыток входа за 10 секунд`, ТО `отправить алерт "Множественные неудачные попытки входа"`.
    *   ЕСЛИ `после нескольких неудачных попыток вход стал успешным`, ТО `отправить алерт "Успешный вход после брутфорса"`.
    *   ЕСЛИ `исходящий трафик с хоста > 25 МБ` (при политике запрета), ТО `отправить алерт "Потенциальная кража данных"`.

### Как создается правило: Примеры на основе Event ID

**Пример 1: Очистка журналов событий**
Злоумышленники часто пытаются стереть логи, чтобы скрыть следы. В Windows за это отвечает Event ID 104.
*   **Правило:** `(Source = WinEventLog) AND (EventID = 104)`
*   **Результат:** Оповещение "Event Log Cleared".

**Пример 2: Выполнение команды WHOAMI**
Злоумышленники часто используют команду `whoami` для проверки своих привилегий. За запуск процессов отвечает Event ID 4688.
*   **Правило:** `(Source = WinEventLog) AND (EventID = 4688) AND (NewProcessName contains "whoami")`
*   **Результат:** Оповещение "WHOAMI command Execution DETECTED".

> **Важно:** Правила опираются на значения конкретных полей. Именно поэтому необходима **нормализация** данных — чтобы поле `NewProcessName` существовало и было заполнено корректно.

## 6. Расследование инцидента

При срабатывании оповещения аналитик проводит следующие шаги:

1.  **Анализ:** Изучает дашборды и события, связанные с алертом.
2.  **Верификация:** Проверяет, соответствует ли событие правилу, и является ли оно злонамеренным на самом деле.
3.  **Классификация:**
    *   **Ложное срабатывание (False Positive):** Требуется донастройка правила, чтобы исключить подобные случаи в будущем.
    *   **Истинное срабатывание (True Positive):** Требуются дальнейшие действия.
4.  **Реагирование:**
    *   Проведение глубокого расследования.
    *   Связь с владельцем актива для уточнения деталей.
    *   **Подтверждение угрозы:**
        *   Изоляция зараженного хоста.
        *   Блокировка подозрительного IP-адреса.
