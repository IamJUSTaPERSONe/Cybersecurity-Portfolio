# Анализ фишинговых писем: от основ до расследования

## Введение

Каждое фишинговое письмо начинается с простого — с адреса электронной почты. Понимание его структуры помогает замечать подсказки, на которые рассчитывают злоумышленники.

> **История**: Электронная почта в современном виде появилась в 1970-х годах в ARPANET. Рэй Томлинсон ввел символ `@`, чтобы отделить имя пользователя от системы.

---

## Структура электронного адреса

Пример: `david@tryhackme.com`

| Компонент       | Описание                                                  |
|----------------|-----------------------------------------------------------|
| Имя пользователя (`david`) | Идентифицирует почтовый ящик на сервере                 |
| Символ `@`      | Разделяет имя пользователя и домен                        |
| Доменное имя (`tryhackme.com`) | Указывает на почтовый сервер получателя          |

### Аналогия с почтовым адресом

- **Домен** = улица или многоквартирный дом
- **Имя пользователя** = конкретный человек или почтовый ящик

---

## Сетевые протоколы электронной почты

| Протокол | Назначение |
|----------|-------------|
| **SMTP** (Simple Mail Transfer Protocol) | Отправка писем |
| **POP3** (Post Office Protocol) | Загрузка писем на одно устройство |
| **IMAP** (Internet Message Access Protocol) | Синхронизация писем на нескольких устройствах |

### Сравнение POP3 и IMAP

| Характеристика | POP3 | IMAP |
|----------------|------|------|
| Хранение писем | На одном устройстве | На сервере |
| Доступ с нескольких устройств | ❌ | ✅ |
| Синхронизация | ❌ | ✅ |
| Удаление с сервера | Обычно после загрузки | Вручную |

---

## Путь письма: от отправителя к получателю

1. **Отправка** – клиент отправителя использует SMTP для передачи письма на свой почтовый сервер
2. **DNS-запрос** – сервер отправителя запрашивает DNS, чтобы найти почтовый сервер получателя
3. **Ответ DNS** – DNS возвращает адрес сервера получателя
4. **Доставка** – письмо передается через Интернет на сервер получателя
5. **Проверка** – клиент получателя подключается к своему почтовому серверу
6. **Получение** – письмо загружается (POP3) или синхронизируется (IMAP)

---

## Анатомия электронного письма

Письмо состоит из двух частей:

1. **Заголовок (Header)** – метаданные (отправитель, серверы, дата и т.д.)
2. **Тело (Body)** – само сообщение (plain text или HTML)

### Основные поля заголовка

| Поле | Описание |
|------|-----------|
| `From:` | Адрес отправителя |
| `To:` | Адрес получателя |
| `Reply-To:` | Адрес для ответов (опционально) |
| `Subject:` | Тема письма |
| `Date:` | Время и дата отправки |

> Просмотр **исходного кода сообщения** (Ctrl+U в Thunderbird) дает полную картину, включая IP-адреса и технические детали.

### Работа с вложениями в исходном коде

Пример заголовков вложения:

Content-Type: application/pdf
Content-Disposition: attachment; filename="invoice.pdf"
Content-Transfer-Encoding: base64
 
Само вложение закодировано в **base64**. Расшифровать можно через [CyberChef](https://gchq.github.io/CyberChef/).

## Типы вредоносных писем

| Тип | Описание |
|-----|-----------|
| **Спам** | Нежелательные массовые рассылки |
| **Фишинг** | Выдача себя за доверенное лицо с целью кражи данных |
| **Специализированный фишинг** | Целенаправленная атака на конкретного человека/организацию |
| **Whaling** | Фишинг на топ-менеджеров (CEO, CFO) |
| **Smishing** | Фишинг через SMS |
| **Vishing** | Фишинг через голосовые звонки |

## Анатомия фишингового письма: признаки

| Признак | Пример |
|---------|--------|
| Поддельный адрес отправителя | `noreply@microsof.com` |
| Срочность | "Ваша учетная запись будет заблокирована через 24 часа" |
| Имитация бренда | Использование логотипов и цветов известной компании |
| Ошибки грамматики | Неестественные формулировки |
| Общее обращение | "Уважаемый клиент" вместо имени |
| Скрытые/сокращенные ссылки | `bit.ly/secure-login` |
| Вредоносные вложения | `invoice.pdf.exe` |

## Правила безопасного анализа

**Никогда не кликайте по подозрительным ссылкам и вложениям!**

### Дефангинг (Defanging) — обезвреживание ссылок

Превращаем кликабельные элементы в безопасные:

| Исходный | Обезвреженный |
|----------|----------------|
| `http://www.suspiciousdomain.com` | `hxxp[://]www[.]suspiciousdomain[.]com` |
| `user@domain.com` | `user[@]domain[.]com` |

## Дополнительно: BEC (Business Email Compromise)

> **Компрометация деловой электронной почты** — атака, при которой злоумышленник получает доступ к легитимному внутреннему ящику и использует его для мошенничества.

---

## Резюме

1. Понимание структуры адреса и протоколов помогает анализировать подозрительные письма.
2. Исходный код письма дает максимум информации (заголовки, вложения, реальные ссылки).
3. Фишинг использует социальную инженерию и технические подделки.
4. Всегда применяйте **дефангинг** при работе с потенциально опасными ссылками.
